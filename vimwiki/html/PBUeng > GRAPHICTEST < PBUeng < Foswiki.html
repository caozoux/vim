Graphics accelerators test

1 Graphics accelerators introdution

  • As the demand for better graphics increased, hardware manufacturers created a way to decrease the amount of CPU time required to fill the framebuffer. This is commonly called a "graphics accelerator" in the Unix world.
  • Common graphics drawing commands (many of them geometric) are sent to the graphics accelerator in their raw form. The accelerator then rasterizes the results of the command to the framebuffer. This method can save from thousands to millions of CPU cycles per command, as the CPU is freed to do other work.
  • While early accelerators focused on improving the performance of 2D GUI systems, most modern accelerators focus on producing 3D imagery in real time. A common design is to send commands to the graphics accelerator using a library such as OpenGL?. The OpenGL? driver then translates those commands to instructions for the accelerator's graphics processing unit (GPU). The GPU uses those microinstructions to compute the rasterized results. Those results are bit blitted to the framebuffer. The framebuffer's signal is then produced in combination with built-in video overlay devices (usually used to produce the mouse cursor without modifying the framebuffer's data) and any analog special effects that are produced by modifying the output signal. An example of such analog modification was the anti-aliasing technique used by the 3dfx Voodoo cards. These cards add a slight blur to output signal that makes aliasing of the rasterized graphics much less obvious.
  • Popular manufacturers of 3D graphics accelerators are Nvidia and ATI Technologies

2 What is the DRI

  • The technical description from the DRI website describes it as follows:
  • The direct rendering infrastructure, also known as the DRI, is a framework for allowing direct access to graphics hardware in a safe and efficient manner. It includes changes to the X server, to several client libraries, and to the kernel. The first major use for the DRI is to create fast OpenGL? implementations.
  • What this means is that the DRI is a framework for allowing programs to speak directly to your graphics card. For example, games or graphics programs can send commands directly to your card and let it perform fast, hardware-accelerated rendering producing high-quality visuals. At the same time your CPU is free to do other work. This is known as direct rendering.
  • Without the DRI programs have to perform the rendering using software. This ties up your CPU and results in much lower framerates since software is slower than hardware. This is know as indirect rendering.

3 gfx on AtomE6xx? platform

  • purpose:
    • This design aims at integrating a xorg working environment on the tunnel creek platform, under this environment, 2D and 3D drivers should be able to support GMA500/600 (PowerVR? SGX535) GPU and work well both for 2D Xapps and 3D Xapps
  • scope:
    • To implement a HW accelerated xorg working environment on the tunnel creek platform, we should do some work both for Linux kernel and for user space.
    • Kernel: Integrate agpgart and DRM driver from the latest IEMGD package
    • user space: Integrate compatible version 3D lib Mesa, integrate compatible version DRM lib libdrm, integrate compatible version xorg server xorg-server, integrate various xorg-server needed packages, integrate 2D driver emgd_drv.so from IEMGD package, integrate dri 3D driver emgd_dri.so from IEMGD package, integrate vaapi lib libva from IEMGD package, integrate several PVR relating libs from IEMGD
  • Architecture Overview
    • GMA500/600 GPU is based on the PowerVR? SGX 535 core, is licensed from Imagination Technology and produced by Intel. We can only get its driver from IEMGD package, the kernel DRM driver is source code while the userspace 2D and 3D driver is binary only. Following diagram list key components (GPU relating) for a xorg working environment on the tunnel creek platform

  • General
    • To support GMA500/600 GPU in the xorg environment, we need:
      • Support following drivers in the linux kernel:
        • Intel agpgart: allocate system memory and remap it for GPU usage. IEMGD package provides GMA500/600 specific codes for intel agpgart driver, it is bond into DRM driver and is not a separate one.
        • Intel DRM EMGD driver: provides synchronized and security access to the graphics hardware, provides encapsulated DMA queue for render command and data to the graphics hardware. IEMGD package provides source code for this driver.
      • Adding the following key packages in the userspace:
        • Xorg-server: is the core of X window system, the version must match the 2D driver emgd_drv.so.
        • emgd_drv.so : is the xorg 2D driver for Intel GMA 500/600 series of GPU.
        • libdrm: provides core library routines for the X Window System to directly interface with video hardware using the Linux kernel's DRM driver.
        • Mesa: provides 3D hardware drivers and 3D implementation libraries, i.e. OpenGL?, OpenGL? ES.
        • Several PVR relating libs: these several libs are needed to support emgd_drv.so and emgd_dri.so.
        • Libva: is video acceleration API, It enables and provides access to graphics hardware (GPU) acceleration for video processing, i.e. video decoding, video encoding, subpicture blending and rendering.

4 gfx on SandyBridge platform

  • purpose:
    • This design aims at integrating a xorg working environment on the SandyBridge platform, under this environment, 2D and 3D drivers should be able to support SandyBridge GPU and work well both for 2D Xapps and 3D Xapps
  • scope:
    • To implement a HW accelerated xorg working environment on the SandyBridge platform, we should do some work both for Linux kernel and for user space.
    • Kernel: backport latest Intel AGP driver, backport latest Intel DRM i915 driver with KMS functions enabled
    • user space: Integrate latest xorg 2D driver xf86-video-intel, integrate latest 3D lib and driver Mesa, integrate latest video api library libva, integrate latest DRM lib libdrm, integrate latest xorg server xorg-server, integrate various xorg-server needed packages

    • SandyBridge GPU is the latest product of Intel, is called generation 6 Intel GMA (Graphic Media Accelerator), if we want to support this GPU both in 2D and 3D drivers, we need to refer to the intellinuxgraphics.org and use the correct version both for kernel and for xorg packages. In general, we should choose latest version for those hardware relating packages. Following diagram list key components (GPU relating) for a xorg working environment on the SandyBridge platform

  • General
    • To support SandyBridge GPU in the xorg environment, we need:
      • Integrate following drivers in the linux kernel:
        • Intel agpgart: allocate system memory and remap it for GPU usage.
        • Intel drm i915 driver with KMS enabled: provides synchronized and security access to the graphics hardware, provides encapsulated DMA queue for render command and data to the graphics hardware, provides display mode setting in the kernel.
      • Adding the following key packages in the userspace:
        • Xorg-server: is the core of X window system.
        • Xf86-video-intel: is the xorg 2D driver for Intel GMA series(except GMA 500/600) of GPU.
        • libdrm: provides core library routines for the X Window System to directly interface with video hardware using the Linux kernel's DRM driver
        • Mesa: provides 3D hardware drivers and 3D implementation libraries, i.e. OpenGL?, OpenGL? ES.

5 gfx on Ti omap3 platform

  • purpose:
    • This design aims at integrating a xorg working environment on the TI omap3 platform, under this environment, OpenGL? ES 1.1, OpenGL? ES 2.0 and OpenVG? 1.0 3D/2D applications can be accelerated on the PowerVR? SGX 530 GPU and can runs well both in raw console and in xorg window app.
  • scope:
    • To implement a HW accelerated xorg working environment on the TI omap3 platform, we should do some work both for Linux kernel and for user space.
    • Kernel: Integrate DRM driver provided in the TI Graphics_SDK package
    • user space: Integrate xorg 2D driver xf86-video-omapfb from http://git.pingu.fi, integrate latest xorg server xorg-server (1.9.0 or above), integrate various xorg-server needed packages (xproto, xlib, …), integrate optimized OpenGL? ES libs, OpenVG? libs and libEGL optimized for PowerVR? GPU from Graphics_SDK package, integrate several PowerVR? GPU specific libs from Graphics_SDK package
  • Architecture Overview
    • Ti Omap3 series MCU integrate a GPU which is based on the PowerVR? SGX 530 core. Currently OpenGL? ES 1.1, OpenGL? ES 2.0 and OpenVG? 1.0 are implemented and optimized for this GPU. That is to say we can develop accelerated 3D/2D applications basing on them, but those applications can only run in raw console, can’t work in the xorg environment. To change this situation, we need to integrate xorg environment (xorg-server, xproto, xlibs, …) and xorg 2D driver, and use the optimized libEGL to connect between xorg and accelerated apps. Ti Graphics_SDK provides optimized OpenGL? and OpenVG? libs and headers, optimized libEGL, several PowerVR? specific libs in binary style and kernel DRM driver source code. The xorg 2D driver xf86-video-omapfb is an open source project, we can get its source code from http://git.pingu.fi

  • General
    • To support accelerated OpenGL? ES and OpenVG? apps to run in the xorg environment, we need:
      • Support following drivers in the linux kernel:
        • DRM driver: provides synchronized and security access to the graphics hardware, provides encapsulated DMA queue for render command and data to the graphics hardware. Ti Graphics_SDK package (3_01_00_02, 3_01_00_06, 4_00_00_01) provides source code for this driver.
      • Adding the following key packages in the userspace:
        • Xorg-server: is the core of X window system, because we can get the source code of xorg 2D driver. We can use xorg-server-1.9.0 or above version, when configure, we should add disable-dri and disable-glx
        • Xf86-vdieo-omapfb: is the xorg 2D driver, This driver is not accelerated for PowerVR? GPU but interface for omap3 DSS2 (display subsystem 2)framebuffer driver, that is to say this driver nearly can’t provide 2D acceleration capability. We can get its latest source code from http://git.pingu.fi.
        • OpenGL? ES and OpenVG? libs and headers: provides OpenGL? ES 3D and OpenVG? 2D implementation libraries and headers, basing on these libs and headers, we can develop accelerated 3D/2D appliacations. Ti Graphics_SDK package provides these libs in binary style and those header files.
        • Several PVR relating libs: these several libs (libIMGegl.so, libpvr2d.so, libsrv_um.so, libpvrPVR2D_BLITWSEGL.so, …) are needed to support optimized OpenGL? ES and OpenVG? applicatons. Ti Graphics_SDK package provides these libs in binary style.
        • libEGL: is a interface between OpenGL? ES and OpenVG? to xorg window system. With this lib’s help, we can develop OpenGL? ES and OpenVG? xapps. Ti Graphics_SDK package provides these libs in binary style.

6 External Matrox Graphics MGA G550 PCIe x1 Video Card

    • For some reason, an external PCIe Video Card is necessary to help us test graphics functionality on intel x86 platform. For example, sometimes the board doesn't integrate a built-in video engine or gpu.

  • how to test with fbdev driver
    • 1. configure --with-template=feature/xapps,feature/media_frame
    • 2. enable kernel options. 
        Kernel menuconfig Location:
                -> Device Drivers
                   -> Graphics support
                      -> Support for frame buffer devices
                         -> Matrox acceleration
                            -> G100/G200/G400/G450/G550 support
    • 3. boot target.
    • 4. X -configure
    • 5. vi xorg.conf.new # change Driver to "fbdev".

  • trouble shooting
    • 1. errors like "VESA(0): V_BIOS address 0x9d400 out of range" when using vesa mode
      This is due to VideoBIOS?. You can't use VESA mode.
    • 2. DRM and DRI driver
      Currently there is no kernel driver supporting G550 chip. In user space, xf86-video-mga is not integrated into wrlinux yet. So DRI and DRM are unavaible in wrlinux.
